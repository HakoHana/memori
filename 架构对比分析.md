# 架构对比：现有实现 vs 新架构设计

> 对比对象：当前运行的代码（~4865 行 Python + 19 张表） vs 架构设计.md v2.1
> 分析目的：确认现有代码与新架构的差距，指导实施顺序

---

## 一、总览

| 维度 | 现有实现 | 新架构设计 | 差距 |
|------|---------|-----------|------|
| 核心代码行数 | ~4865 行 | — 待实现 — | — |
| 数据库表 | 19 张（含 FTS 内部表） | 14 张（不含 FTS 内部） | 3 张新表 |
| 配置项 | 14 项 | 20+ 项 | 6+ 新增 |
| 分层 | 无热-温-冷概念 | 明确三层 | **大缺口** |
| 用户身份 | `user_registry`（QQ→名字） | `canonical_users` + `user_identities` | 需扩展 |
| 画像 | `personas/{user_id}.md` 文件 | `user_persona` SQLite 表 | 需迁移 |
| 图谱 | entity + mentions 仅一种边 | 5 种节点 + 4 种边 + co_occur | **大缺口** |
| 日记不可变 | 可覆盖更新 | 只追加，correction 标记 | 需改 |
| 归档 | 无 | 自动冷存储 Markdown | 新模块 |
| FTS5 | 在 memory_atoms 上 | 在 diary_entries 上 | 需迁移 |

## 二、逐模块对比

### 2.1 图谱（L4）— 差距最大的模块

| 特性 | 现有实现 | 新架构 | 是否已有 | 工作量 |
|------|---------|--------|---------|--------|
| `mentions` 边 | ✅ entity → (target=0) | ✅ entity → diary_id | ⚠️ target 指向空 | 极小 |
| co_occur 边 | ❌ | ✅ | 新表 + 写入逻辑 | 中 |
| 日记节点 | ❌ | ✅ node_type="diary" | 新增 | 小 |
| user 节点 | ❌ | ✅ node_type="user" | 新增 | 小 |
| 邻居查询 | ❌ | ✅ get_neighbors() | 新增 | 小 |
| 图→表降级决策 | ❌ | ✅ 关键词匹配→co_occur | 新增 | 中 |
| co_occur 查询 | ❌ | ✅ O(1) 查表 | 新增 | 小 |

**现有代码路径**：
```
capturer.capture()
  → on_atoms_created → graph_engine.index_diary(diary_id, content, entities)
      → extract_wikilinks → ["Hako"]
      → upsert_nodes → entity 节点
      → add_edge_by_ids(source=entity_id, target=0, relation="mentions")
                                          ↑
                                      target=0 是空的！
```

**改后路径**：
```
capturer.capture()
  → on_atoms_created → graph_engine.index_diary(diary_id, content, entities)
      → 1. 创建日记节点 (node_type="diary", node_key=f"diary:{diary_id}")
      → 2. 解析 [[实体]] → 创建 entity/user 节点
      → 3. add_edge(source=entity_id, target=diary_id, "mentions")
                                           ↑
                                      真实 diary_id ✅
      → 4. 本日记所有实体两两 → entity_cooccur.upsert(count++)
```

## 四、总结

```
现有实现好的地方：
  ✅ L1/L2 核心数据模型已完备
  ✅ 图谱 [[实体]] 自动建边已有
  ✅ 用户名字提取和存储已做
  ✅ FTS5 全文搜索
  ✅ atomic_facts 去重+强化
  ✅ diary_fact_links 多对多关联

现有实现缺失（按严重度）：
  🔴 co_occur 边 + 实体关系查询（P0，最常用）
  🔴 日记节点 + mentions 指向真实 diary_id（P0）
  🟡 用户身份扩展（P1）
  🟡 画像增量更新（P1）
  🟡 FTS5 迁移到 diary_entries（P2）
  🟢 只追加不可变（P2）
  🟢 归档 + 热温冷分层（P3）

新架构比现有实现多：
  3 张新表（canonical_users, user_identities, user_persona）
  1 张新共现表（entity_cooccur）
  6+ 个新配置项
  1 个归档模块
```

要不要从 **P0 图谱补全**开始写代码？三个改动都很小，加起来不到 50 行，但图谱效果会有明显提升。
