# Hana 记忆系统架构设计

> 版本：v2.1 — 2026-06-08
> 理念：**日记是唯一真相源，原子事实是指针，用户身份是图谱，图谱是推理网**

---

## 第一章：设计哲学

### 1.1 核心原则

| 原则 | 含义 | 违反代价 |
|------|------|---------|
| **单一可信源** | L1 日记是唯一真相，L2/L3/L4 都是派生视图 | 数据不一致，无法信任 |
| **只追加不修改** | 日记写入后不可变，更正→新条目加 `correction` 标记 | 历史被篡改，不可追溯 |
| **可重建** | 所有派生数据能从 L1 恢复 | 算法升级后旧数据成垃圾 |
| **热-温-冷分层** | 自动分层，平衡成本与速度 | 全量 SQLite 迟早撑爆 |
| **身份可辨** | 每人一个规范 UID，跨平台关联 | 同一用户在不同平台有多份画像 |

### 1.2 分层模型

```
L0 原始对话 ──→ messages 表 → 短期保留，自动过期
                      │
L1 日记 ──→ diary_entries（真相源，不可变）
    │                ├── SQLite（热/温）
    │                └── Markdown（冷）
    │
    ├──→ L2 原子事实（atomic_facts + diary_fact_links）
    │        派生视图，可从 L1 重建
    │
    ├──→ L3 用户画像（user_persona）
    │        派生视图，可从 L1+L2 重建
    │
    └──→ L4 知识图谱（graph_nodes + graph_edges）
             派生视图，可从 L1 [[实体]] 重建
```

### 1.3 用户身份体系

```
平台 ID（可变，多个）          规范 UID（唯一，不变）        数据归属
────────────────────────────────────────────────────────────
qq:2398604399          ──→
wx:hako123             ──→    u_8a3f1c2b（UUID）  ──→  diary_entries
tg:tg_hako             ──→                                   atomic_facts
                                                              user_persona
                                                              graph_nodes
```

## 第二章：存储层

### 2.1 热-温-冷策略

| 层 | 范围 | 存储 | 检索行为 |
|---|------|------|---------|
| **热** | ≤7 天 | SQLite | 自动注入 LLM 上下文 |
| **温** | 8~90 天，importance > 0.1 | SQLite | 按需检索（提问触发） |
| **冷** | >90 天 或 importance < 0.1 | Markdown 文件 | 不自动检索，需显式指令 |

配置项：

```yaml
memory:
  hot_days: 7
  warm_days: 90
  cold_importance_threshold: 0.1
```

### 2.2 所有表 Schema

#### canonical_users（规范用户 ID）

```sql
CREATE TABLE canonical_users (
    uid TEXT PRIMARY KEY,                  -- u_8a3f1c2b（UUID，终身不变）
    primary_name TEXT,                     -- 最终显示名
    identity_confidence REAL DEFAULT 0.3,  -- 身份确定性 0~1
    created_at REAL,
    updated_at REAL
);
```

#### user_identities（平台 ID → UID）

```sql
CREATE TABLE user_identities (
    platform_id TEXT PRIMARY KEY,          -- "qq:2398604399"
    uid TEXT NOT NULL REFERENCES canonical_users(uid),
    platform TEXT NOT NULL,                -- "qq" / "wechat" / "telegram"
    display_name TEXT,                     -- 该平台昵称（当时的值）
    first_seen REAL,
    last_seen REAL,
    verified INTEGER DEFAULT 0,           -- 1=已手动确认
    source TEXT DEFAULT 'auto'             -- auto / manual / llm_inferred
);
```

#### diary_entries（L1 日记 — 唯一真相源）

```sql
CREATE TABLE diary_entries (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    uid TEXT,                              -- 规范 UID（Phase 2 后替代 user_id）
    user_id TEXT NOT NULL,                 -- 当前平台 ID（过渡期保留）
    date TEXT NOT NULL,                    -- YYYY-MM-DD
    timestamp REAL NOT NULL,
    content TEXT NOT NULL,                 -- frontmatter + Markdown（唯一真相）
    importance REAL DEFAULT 0.5,           -- MAX(原子重要度)
    mood TEXT,
    topics TEXT,                           -- JSON array
    source TEXT,                           -- qq / wechat / telegram
    fact_extracted INTEGER DEFAULT 0,      -- 0=待提取 1=成功 -1=失败
    fact_retry_count INTEGER DEFAULT 0,    -- 当前重试次数
    archived INTEGER DEFAULT 0,            -- 0=热/温 1=已归档冷
    correction INTEGER DEFAULT 0,          -- 0=正常 1=更正条目
    created_at REAL,
    updated_at REAL
);
CREATE INDEX idx_diary_uid_time ON diary_entries(uid, timestamp);
CREATE INDEX idx_diary_archived ON diary_entries(archived);
CREATE INDEX idx_diary_fact_status ON diary_entries(fact_extracted);
```

**FTS5 索引**（在 diary_entries.content 上，唯一的全文检索引擎）：

```sql
CREATE VIRTUAL TABLE diary_fts USING fts5(
    content, content=diary_entries,
    tokenize='unicode61'
);
```

注意：原子事实检索不走 FTS5，通过结构化字段直接匹配。

```sql
CREATE TABLE graph_nodes (
```

#### graph_edges（图谱边）

```sql
CREATE TABLE graph_edges (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    target_node_id INTEGER NOT NULL,
    relation_type TEXT NOT NULL,            -- mentions / co_occur / relates_to
    source_memory_id INTEGER NOT NULL,      -- 来源日记 ID
```

## 第三章：图谱（L4）实现

### 3.1 节点体系

| node_type | 例子 | 来源 | 说明 |
|-----------|------|------|------|
| `entity` | Hako, GPT-SoVITS | `[[实体]]` | 通用实体 |
| `user` | u_8a3f1c2b | 自动识别（匹配 user_identities） | 规范用户节点 |
| `diary` | diary:61 | 写日记时创建 | 每篇日记一个节点 |
| `topic` | 技术讨论 | LLM 分类或 frontmatter.topics | 话题聚类 |
| `emotion` | happy, sad | frontmatter.mood | 情绪标签 |
| `date` | 2026-06 | 日记日期 | 时间维度 |

### 3.2 边体系

| 边类型 | 方向 | 来源 | 说明 |
|--------|------|------|------|
| `mentions` | entity → diary | `[[实体]]` | 日记提到了实体 |
| `co_occur` | entity ↔ entity | 共现统计 | 两个实体在同一日记出现 |
| `relates_to` | entity ↔ entity | LLM 推断（离线） | 语义关联 |
| `same_user` | user ↔ platform_id | 身份合并 | 同一用户的不同平台 ID |

### 3.3 写入路径

```
capturer.capture() → 写日记
  │
  └─→ graph_engine.index_diary(diary_id, content, entities)
        │
        ├─ 解析 [[Hako]] → ["Hako"]                            ← extract_wikilinks()
        ├─ 合并 atoms.entities → ["Hako", "GPT-SoVITS"]
        │
        ├─ 创建日记节点                                         ← 新增 1.2
        │   graph_store.upsert_nodes([
        │     GraphNode(node_key="diary:61", node_type="diary")
        │   ])
        │
        ├─ 创建实体节点                                        ← 已有
        │   graph_store.upsert_nodes([
        │     GraphNode(node_key="entity:hako", node_type="entity"),
        │     GraphNode(node_key="entity:gpt-sovits", node_type="entity"),
        │   ])
        │
        ├─ 如果实体匹配 user_identities → node_type="user"     ← 新增 1.3
        │
        ├─ 创建 mentions 边（实体 → 日记）                     ← 修复 1.2
        │   add_edge(source=entity_id, target=diary_id,
        │           relation_type="mentions")
        │
        └─ 更新 co_occur 计数                                  ← 新增 1.1
            entity_cooccur.upsert(WHERE 当前日记所有实体两两组合)
```

### 3.4 查询路径

```
用户问题
  │
  ├─ 含关系词？("关系"/"共同"/"关联"/"对比"/"除了…还")
  │    是 → 走图查询
  │    否 → 走表查询（FTS5 + atomic_facts）
  │
  ├─ 表查询结果充分？（前 3 条相关性 > 0.6）
  │    否 → 降级到图查询
  │
  └─ 图查询策略：
        ├─ 共现 → SELECT * FROM entity_cooccur WHERE entity_a_id=? ORDER BY count DESC
        │          O(1)，直接查表
        ├─ 一跳邻居 → SELECT * FROM graph_edges WHERE source_node_id=? OR target_node_id=?
        │          <5ms，索引覆盖
        ├─ 两跳路径 → BFS 两层
        │          20~100ms，仅离线/非实时
        └─ 语义关联 → LLM 推断 → 离线，不参与对话
```

### 3.5 图结果注入策略

```
图查询结果 → 不自动注入 LLM
  │
  ├─ 结果作为可选项，LLM 决定是否引用
  ├─ 如需注入，只注入 1~2 条路径
  ├─ 每条路径 ≤ 5 个节点
  └─ 总 token ≤ 200
```

## 第四章：核心数据流

### 4.1 完整写入路径

```
用户消息
  │
  ├─→ main.py: on_llm_request()
  │      ├─ 提取 sender_id + sender_name
  │      ├─ 写入 user_identities（新平台 ID 时）
  │      ├─ 写入 messages（L0）
  │      └─ 记忆注入（召回 → 拼入上下文）
  │
  └─→ consolidation_manager.on_message(user_id, text)
         │
         ├─ 计数 → 判断是否触发 L1
         │
         └─ capturer.capture()（触发时）
                │
                ├─ 1. should_capture() → LLM 判断
                ├─ 2. _write_diary() → LLM 写日记正文
                ├─ 3. build_diary_content() → 封装 frontmatter
                ├─ 4. diary_store.append() → 写入 diary_entries（fact_extracted=0）
                │
                ├─ 5. 异步 LLM 提取原子
                │      ├─ 成功 → atomic_facts（去重/强化）
                │      │        → diary_fact_links（关联）
                │      │        → diary_entries.fact_extracted=1
                │      ├─ 重试（最多 3 次）
                │      └─ 失败 → diary_entries.fact_extracted=-1
                │
                ├─ 6. 同步图谱更新
                │      ├─ 创建 diary 节点
                │      ├─ 创建 entity/user 节点
                │      ├─ 创建 mentions 边
                │      └─ 更新 co_occur
                │
                ├─ 7. diary.importance = MAX(原子重要度)
                │
                └─ 8. 检查 L3（每 N 篇日记触发画像增量更新）
```

### 4.2 读取路径

```
用户提问
  │
  │  意图分类（关键词匹配）
  ├── "关系"/"共同"/"关联" → 图查询
  │     ├─ co_occur 表（O(1)）
  │     └─ 一跳邻居（<5ms）
  │
  └── 其他 → 表查询
        ├─ 热区（≤7 天）：自动检索 L1+L2+L3
        ├─ 温区（8~90 天）：按需检索
        └─ 冷区（>90 天）：不检索
              │
              ▼
        排序 → 取 TopK → 注入 LLM
```

### 4.3 归档路径

```
每日定时器 → 扫描 diary_entries
  │
  ├─ timestamp > 90 天 AND importance < 0.1
  │     ├─ 导出 Markdown → memory_archive/{uid}/YYYY/MM/YYYY-MM-DD.md
  │     │  同一天多条日记合并到一个文件，用 --- 分隔
  │     ├─ content 保留前 200 字摘要
  │     └─ archived=1, archived_file_path=...
  │
  ├─ persona 全量重建时
  │     └─ 导出 → memory_archive/{uid}/persona/YYYY-MM-DD_v{version}.md
  │
  └─ entity_cooccur 清理
        └─ 删除 count=1 且 last_updated > 90 天的条目
```

## 第五章：模块接口

### 5.1 GraphEngine（L4）

```python
class GraphEngine:
    """图谱引擎 — 建节点、建边、共现统计"""

    async def index_diary(
        self,
        diary_id: int,
        content: str,
        user_id: str = "",
        entities: list[str] | None = None,
    ):
        """写日记时调用：创建 diary 节点 + 实体节点 + mentions 边 + co_occur"""
        # 1. 解析 [[实体]]
        # 2. 创建 diary 节点 (node_type="diary", node_key=f"diary:{diary_id}")
        # 3. 创建/更新实体节点
        #    - 如果实体名匹配 user_identities → node_type="user"
        #    - 否则 → node_type="entity"
        # 4. 创建 mentions 边: entity → diary
        # 5. 更新 co_occur: 本日记所有实体两两组合 → count++

    async def get_cooccurring(
        self, entity_name: str, k: int = 10
    ) -> list[tuple[str, int]]:
        """获取与某实体最常共现的实体（O(1)查表）"""

    async def get_neighbors(
        self, entity_name: str, relation_type: str = ""
    ) -> list[dict]:
        """获取实体的 k=1 邻居（实体→日记→其他实体）"""

    async def rebuild_from_scratch(self, confirm: bool = False):
        """全量重建图谱：遍历所有 diary_entries → 重新解析 [[实体]] → 建所有节点边"""
```

## 第六章：运行时决策流程

```
用户问题
  │
  ├─ 关键词匹配（"关系"/"共同"/"关联"/"对比"/"除了…还"）
  │    是 → intent = "relation" → 走图
  │    否 → intent = "fact" → 走表
  │
  ├─ 表查询 → FTS5 + atomic_facts
  │    结果充分？（Top3 相关性 > 0.6）
  │      是 → 直接回答
  │      否 → 降级到图查询（后置）
  │
  └─ 图查询
        ├─ co_occur → O(1) → 可实时 ✅
        ├─ 一跳邻居 → <5ms → 可实时 ✅
        ├─ 两跳 BFS → 20~100ms → 仅离线 ⚠️
        └─ 语义关联 → LLM → 仅离线 ❌
              │
              ▼
        图结果不自动注入 LLM
        ├─ 作为可选项，LLM 决定是否引用
        ├─ 如需注入 ≤ 1~2 条路径，≤ 5 节点/条，≤ 200 tokens
        └─ LLM 可反问："我发现 Hako 和 GPT-SoVITS 可能在同一个话题中出现过，需要展开吗？"
```
