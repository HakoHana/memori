# Hana 记忆系统架构设计 v3.1

> 版本：v3.1 — 2026-06-10
> 基于 v0.3.0 代码实际结构，替代 v3.0

---

## 一、概述

Memory 是一个**日记式长期记忆插件**，为 AstrBot 提供以下能力：

- **日记式记忆**：LLM 以第一人称写日记，记录重要对话时刻
- **原子事实**：从日记提取结构化事实，支持全文检索
- **知识图谱**：自动构建实体关联图，可视化浏览
- **用户画像**：长期沉淀的用户特征画像
- **WebUI Dashboard**：记忆管理、图谱可视化、日记编辑

总代码量约 7300 行 Python，纯 SQLite 存储，零外部中间件依赖。

## 二、数据流总览

```
消息事件
    │
    ├─ main.py: on_llm_request
    │     ├─ memory_injector ── 召回记忆 → 注入 LLM 上下文
    │     └─ conversation_store ── 存储原始对话
    │
    ├─ main.py: on_message
    │     ├─ 热缓存写入（HotCache，零 I/O）
    │     └─ consolidation_manager ── 消息计数 → 触发判断
    │             └─ 阈值到达 → warm_processor.enqueue()
    │
    └─ main.py: on_llm_response
          └─ conversation_store ── 存储 Bot 回复（[Bot: Hana]: 格式）

╔══════════════════════════════════════════╗
║        warm_processor (后台队列)          ║
║  asyncio.Queue → _process_one()          ║
║    │                                     ║
║    ├─ 取完整对话 ctx ← conversation_store║
║    ├─ Judge ── LLM 判断是否值得记          ║
║    ├─ Capture ── 写日记 + 提取原子事实     ║
║    ├─ GraphEngine ── 图谱索引             ║
║    └─ PersonaEngine ── 画像增量更新        ║
╚══════════════════════════════════════════╝

存储层：
  conversation_store → diary_store → atomic_facts + graph_store
       (L0 原始对话)    (L1 日记)     (L2 原子)      (L4 图谱)
                            │
                      persona_engine (L3 画像)
```

## 四、身份体系

系统维护**双层身份映射**，实现"内部用 UID，LLM 只看昵称"：

```
┌──────────────────────────────────────────────┐
│              存储层（内部）                     │
│                                                │
│  canonical_users(uid, primary_name)            │
│    ├─ user_identities(platform_id → uid)       │
│    │    "qq:2398604399" → "Hako"              │
│    │    "discord:xxx"   → "Hako#1234"         │
│    │    "wechat:yyy"    → "Hako"              │
│    │                                           │
│    └─ bot_hana  ← 启动时 init_bot_identity()  │
│         primary_name: "Hana"                   │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│               LLM 层（外部）                   │
│                                                │
│  [昵称]: message                 ← 用户消息   │
│  [Bot: 昵称]: message            ← Bot 消息   │
│                                                │
│  只透露昵称，不暴露 QQ 号/平台 ID              │
└──────────────────────────────────────────────┘
```

**关键设计**：
- `ensure_canonical_user()` 将多平台身份映射到统一 UID
- `get_related_user_ids()` 检索时通过 UID 查所有关联平台
- Bot 身份 `bot_hana` 在 `MemoryCore.initialize()` 中注册
- 整理流水线从 `conversation_store` 取完整对话，Bot 消息标注 `[Bot: Hana]:`
- Prompt 模板格式说明与数据实际格式一致

## 五、三层缓存架构

```
     ┌─────────────────────────────┐
     │  热缓存 (HotCache)           │ deque 内存，每用户 ≤20 条
     │  格式: [Hako]: / [Bot:Hana]:│ Retriever.get_recent_context() 优先读
     │  命中即返回，零 I/O           │
     └─────────┬───────────────────┘
               │ 未命中
     ┌─────────▼───────────────────┐
     │  温层 (WarmProcessor)        │ asyncio.Queue → Judge → Capture → L3
     │  异步后台，不阻塞回复         │ 触发条件：消息数/时间间隔/空闲超时
     │  完整对话来自 conversation    │ 入队时传完整 ctx（含 Bot 回复）
     └─────────┬───────────────────┘
               │ 整理完成
     ┌─────────▼───────────────────┐
     │  冷层 (Archiver)             │ >90天 且 importance<0.1 → Markdown
     │  定时扫描                    │ memory_archive/{uid}/YYYY/MM/
     └─────────────────────────────┘
```

## 六、核心表结构

### sessions / messages（L0 原始对话）

```sql
sessions(session_id PK, user_id, platform, created_at, last_active_at, message_count)
messages(id PK AUTO, session_id FK, role, content, sender_id, sender_name, timestamp)
```

**重点**：`role` 区分 `user` / `assistant`，`sender_name` 存显示名，`sender_id` 存平台 ID。

### diary_entries（L1 日记 — 事实真相源）

```sql
diary_entries(
    id, uid, user_id, date, timestamp, content,
    importance, mood, topics,
    fact_extracted, fact_retry_count,
    archived, correction
)
```

### atomic_facts（L2 原子事实 — 全局去重）

```sql
atomic_facts(
    id, uid, user_id,
    content UNIQUE,                                        -- 全局去重键
    atom_type, importance, confidence,
    source_count, fact_version
)
```

### diary_fact_links（L1↔L2 多对多）

```sql
diary_fact_links(
    diary_id, fact_id,
    snippet, importance_in_diary,
    PRIMARY KEY (diary_id, fact_id)
)
```

### user_persona（L3 画像）

```sql
user_persona(
    uid PRIMARY KEY,
    summary, full_markdown,
    known_ids, primary_name, identity_confidence,
    tier,                                                  -- core/active/occasional/new
    version, last_full_update, last_incremental_update
)
```

### graph_nodes / graph_edges（L4 知识图谱）

```sql
graph_nodes(id, node_key UNIQUE, node_type, value, canonical_value, metadata)
-- node_type: entity / user / diary / topic / emotion / date

graph_edges(id, edge_key UNIQUE, semantic_key,
            source_node_id, target_node_id,
            relation_type, source_memory_id,
            weight, confidence, status)
-- relation_type: mentions / co_occur / relates_to / same_user / on_date / has_meta
```

### entity_cooccur（共现预计算）

```sql
entity_cooccur(
    entity_a_id, entity_b_id,
    count, last_updated,
    PRIMARY KEY (entity_a_id, entity_b_id)
)
```

### canonical_users / user_identities（身份体系）

```sql
canonical_users(uid PRIMARY KEY, primary_name, identity_confidence)
user_identities(platform_id PRIMARY KEY, uid, platform, display_name, verified)
```

### memory_atoms（旧版原子 — 兼容保留）

L2 迁移前使用的旧原子表，FTS5 索引覆盖。新代码优先使用 `atomic_facts`。

## 七、写入流水线（WarmProcessor）

```
WarmProcessor._process_one()
  │
  ├─ 0. 取完整对话上下文
  │       conversation_store.get_recent_context(sid, bot_name="Hana")
  │       → [Hako]: 测试辛苦了
  │         [Bot: Hana]: 谢谢你！我给你带好吃的
  │         [Hako]: 真的吗太好了
  │       (已含 [Bot: 前缀)，_tag_conversation 直接透传
  │
  ├─ 1. judge_provider.should_capture(tagged_ctx) → 是否值得记？
  │       └─ false → 跳过，更新最后整理时间
  │
  ├─ 2. capturer.capture(uid, tagged_ctx, judge_result)
  │       ├─ _write_diary()  → diary_entries 写入 L1
  │       │     └─ _detect_speaker_count() 选 prompt（私聊/群聊）
  │       ├─ _extract_atoms() → LLM 提取原子事实
  │       │     ├─ 去重强化（FTS5 jaccard ≥ 0.6）
  │       │     ├─ 写 atomic_facts（content UNIQUE 全局去重）
  │       │     ├─ diary_fact_links 关联
  │       │     └─ 重要度排序限 top 5
  │       └─ on_atoms_created() → fire-and-forget 图谱索引
  │
  ├─ 3. graph_engine.index_diary()
  │       ├─ 解析实体
  │       ├─ 创建 diary 节点
  │       ├─ 创建/更新 entity/user 节点
  │       ├─ 创建 mentions / has_meta 边
  │       └─ 更新 entity_cooccur
  │
  └─ 4. persona_engine.incremental_update()
          └─ 每 N 篇日记触发 → LLM diff → 更新 user_persona
```

### 关键 identity 设计

```python
# warm_processor.py
async def _tag_conversation(self, user_id, text, sender_name):
    """对话文本已经由 conversation_store 格式化好（含 [Bot: 前缀]），直接透传"""
    if not text or text.startswith("["):
        return text  # ← 完整对话上下文带 [ 开头，原样返回
    return f"[{sender_name}]: {text}"  # ← 单条消息回退加标签
```

### Prompt 格式说明一致性

| Prompt 文件 | 用途 | 最新改动 |
|------------|------|---------|
| `judge.txt` | 判断是否值得记 | 无改动 |
| `diary.txt` | 写日记（单说话者） | 无改动 |
| `private_chat.txt` | 写日记（双人私聊） | 格式说明同步为 `[昵称]:` / `[Bot: 昵称]:` |
| `group_chat.txt` | 写日记（群聊≥2人） | 格式说明同步为 `[昵称]:` / `[Bot: 昵称]:` |
| `atoms.txt` | 从日记提取原子事实 | 强调"禁止使用「用户」" |

## 八、读取流水线（Retriever）

```
用户消息 "宝宝我们的暗号"
  │
  ├─ keyword extraction（jieba 分词）
  │     "宝宝我们的暗号" → jieba.lcut → ["宝宝", "我们", "的", "暗号"]
  │     → 过滤停用词 → ["宝宝", "我们", "暗号"]
  │     → 之前 2-gram: ["宝宝", "宝我", "我们", "们的", "的暗", "暗号"] ❌
  │
  ├─ DualRoute 双路检索
  │     ├─ BM25 文档路（k×3 候选）
  │     │     ├─ 英文/数字 → FTS5 MATCH
  │     │     ├─ 中文       → SQL LIKE %kw%
  │     │     └─ 评分: importance×0.7 + keyword_hit×0.3
  │     │
  │     └─ Graph 图路（k×2 候选）
  │           ├─ keyword → graph_nodes LIKE 匹配
  │           ├─ node → graph_edges 边跳转 → 关联 diary
  │           └─ diary → atomic_facts 取事实
  │
  ├─ RRF 融合（k=60）
  │     score = Σ 1/(60 + rank + 1)
  │     → 按 RRF 得分降序取 top k
  │
  └─ 兜底：双路未命中时按 importance DESC 取
```

### 分层评分策略

```
BM25 匹配分：importance × 0.7 + keyword_hit_score × 0.3
RRF 融合分：Σ 1/(60 + rank + 1)    ← 排名越靠前的列表得分越高
双路权重：  BM25 3x 候选 + Graph 2x 候选 → 融合
```

## 九、分词演进

| 阶段 | 方法 | 示例：宝宝我们的暗号 | 问题 |
|------|------|-------------------|------|
| v2.3 及之前 | Unicode 2-gram | `宝宝, 宝我, 我们, 们的, 的暗, 暗号` | 跨词边界噪音 |
| **v3.1** | **jieba 精确模式** | **`宝宝, 我们, 暗号`** | **无噪音** |

jieba 加载 ~5MB 词典，首次 ~0.4s，后续常驻无额外开销。支持自定义词典（聊天昵称、梗）。

## 十、用户分层（Tier）

| 等级 | 条件 | 预过滤 | LLM 判断 | 画像更新 |
|------|------|--------|---------|---------|
| core | 近 7 天 ≥10 条消息 | 跳过 | 每次 | 每 N 篇 |
| active | 近 30 天 ≥5 条消息 | 跳过 | 每次 | 每 N 篇 |
| occasional | 有消息记录但不达标 | 走过滤 | 仅关键词 | 不更新 |
| new | 无记录 | 走过滤 | 仅关键词 | 不更新 |
