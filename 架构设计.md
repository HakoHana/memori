# 记忆系统架构设计

> 基于 AstrBot 平台的日记式长期记忆插件。
> 设计目标：让 Bot 拥有类人的记忆能力——记得发生过什么、记得用户是谁、记得什么重要。

---

## 一、设计哲学

### 双轨记忆模型

人脑的记忆不是一条一条记录的，而是**故事 + 事实**两层：

```
「今天 Hako 跟我告白了，我好开心」
        ↓
故事层：日记（叙事、情感、时间线）
        ↓
事实层：Hako 向 Bot 告白（结构化、可检索、去重）
```

- **日记给情感**：人可读，有温度，按天归档
- **事实给检索**：机器可搜，全局去重，反复提及自动强化

### 设计原则

| 原则 | 为什么 |
|------|--------|
| **自描述** | 数据不自描述就会腐烂。日记 = frontmatter + Markdown，脱离系统也能读 |
| **去重优先** | 相同的事记一次就够了。重复提及只强化重要度，不产生冗余 |
| **图谱副产物** | 任何额外步骤都会被忘记执行。写日记时顺便建图，零维护成本 |
| **身份可辨** | 群聊里多个人说话，LLM 必须能分清谁是谁 |

## 三、数据模型（由粗到细）

### 3.1 概念模型

```
用户 ── 拥有 ──→ 多篇日记
日记 ── 引用 ──→ 多个事实（多对多）
日记 ── 包含 ──→ 多个原子（旧数据兼容）
用户 ── 绑定 ──→ 一个画像
事实 ── 关联 ──→ 图谱节点
节点 ── 连接 ──→ 节点（通过边）
```

### 3.2 逻辑模型

#### 核心表 1：日记（diary_entries）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| user_id | TEXT | 谁写的（QQ号） |
| date | TEXT | YYYY-MM-DD |
| content | TEXT | **核心**：YAML frontmatter + Markdown |
| topics | TEXT | JSON 数组，["技术","聊天"] |
| sentiment | TEXT | positive / negative / neutral |
| importance | REAL | 0~1，取该日记所有原子重要度的最大值 |
| atom_count | INTEGER | 关联原子数 |
| created_at | REAL | Unix 时间戳 |
| updated_at | REAL | 最后修改时间 |
| status | TEXT | active / archived / forgotten |

**content 结构详解**：

```markdown
---
date: 2026-06-07
mood: 开心
importance: 0.85
topics: ["技术讨论", "bug修复"]
diary_id: 61
atom_count: 12
---

今天和 Hako 讨论了 GPT-SoVITS 的部署方案。
[[Hako]] 担心我在修 bug 时消失。

## 关键事实

- 📌 GPT-SoVITS 需要 Python 3.10 环境
- 💝 Hako 偏好本地部署而非云端
```

- YAML 头：结构化元数据，和 SQL 字段保持同步
- 正文：自然语言叙事
- `[[实体]]`：图谱自动建边的锚点
- SQL 字段是冗余索引，用于快速过滤和排序
- content 是完整表达，用于展示、导出、FTS 搜索

#### 核心表 2：全局事实（atomic_facts）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | |
| content | TEXT **UNIQUE** | 事实文本，全局唯一 |
| atom_type | TEXT | episodic/factual/preference/planned/relational |
| importance | REAL | 0~1，每次被引用 +0.05 |
| confidence | REAL | 0.7 默认 |
| source_count | INTEGER | 被多少篇日记引用 |
| created_at | REAL | |
| updated_at | REAL | |

这个表是整个系统的**去重核心**。

**写入规则**：

```
INSERT INTO atomic_facts (content, ...) VALUES ('Hako告白了', ...)
  ON CONFLICT(content) DO UPDATE
  SET importance = MIN(1.0, importance + 0.05),
      source_count = source_count + 1
```

**查询规则**：

```sql
-- 查某篇日记的所有事实
SELECT af.* FROM atomic_facts af
JOIN diary_fact_links dfl ON af.id = dfl.fact_id
WHERE dfl.diary_id = ?

-- 查某个用户的所有事实
SELECT af.* FROM atomic_facts af
JOIN diary_fact_links dfl ON af.id = dfl.fact_id
JOIN diary_entries d ON d.id = dfl.diary_id
WHERE d.user_id = ?
```

#### 核心表 3：日记-事实关联（diary_fact_links）

| 字段 | 类型 | 说明 |
|------|------|------|
| diary_id | INTEGER | 日记 ID |
| fact_id | INTEGER | 事实 ID |
| importance | REAL | 在该日记语境下的重要度 |
| snippet | TEXT | 日记中对应的原句片段 |

多对多关系表。一条事实被 N 篇日记引用，一篇日记引用 M 条事实。

#### 辅助表 4：旧原子（memory_atoms）

旧系统的原子表，保持向后兼容。新数据同步写入 atomic_facts 和此表，旧数据只从此表读取。

关键字段：`diary_id`（关联日记）、`content`、`importance`、`diary_snippet`

#### 辅助表 5：用户注册表（user_registry）

| 字段 | 类型 | 说明 |
|------|------|------|
| user_id | TEXT PK | QQ号，终身不变 |
| user_name | TEXT | 最新昵称，可改 |
| first_seen_at | REAL | 首次出现时间 |
| last_seen_at | REAL | 最后活跃时间 |
| name_updated_at | REAL | 昵称最后变更时间 |

**设计理由**：
- user_id 是稳定键，用于数据关联
- user_name 是展示名，LLM 上下文里用这个
- 昵称变更不改历史消息的 sender_name（历史快照），只更新此表的当前值

#### 辅助表 6：消息（messages）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | |
| session_id | TEXT | 会话 ID |
| role | TEXT | user / assistant |
| content | TEXT | 消息原文 |
| sender_id | TEXT | 发送者 QQ 号 |
| sender_name | TEXT | **当时的昵称**（历史快照） |
| timestamp | REAL | |

**为什么存 sender_name 而不是每次都去查 user_registry？**
因为用户会改名。存当时的名字，LLM 看到的历史对话就是正确的称呼。改名前后的消息自然地使用不同名字。

#### 辅助表 7：图谱节点（graph_nodes）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | |
| node_key | TEXT UNIQUE | "entity:Hako" |
| node_type | TEXT | entity / topic / atom_type |
| value | TEXT | 显示值 |
| canonical_value | TEXT | 归一化值（小写去空格） |
| metadata | TEXT | JSON |

#### 辅助表 8：图谱边（graph_edges）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | |
| edge_key | TEXT UNIQUE | 去重键 |
| source_node_id | INTEGER | 源节点 |
| target_node_id | INTEGER | 目标节点 |
| relation_type | TEXT | mentions / co_occur |
| source_memory_id | INTEGER | 来源原子 ID |
| weight | REAL | 权重 |

#### 辅助表 9：会话状态（consolidation_state）

| 字段 | 类型 | 说明 |
|------|------|------|
| user_id | TEXT PK | |
| msg_count | INTEGER | 累计消息数 |
| warmup_threshold | INTEGER | 暖启动阈值 |
| last_consolidated_at | REAL | 上次整理时间 |
| last_diary_date | TEXT | 上次写日记日期 |
| diary_count | INTEGER | 日记总数 |
| l1_retry_count | INTEGER | 重试计数 |

### 4.2 L1 整理流水线（最复杂、最关键）

```
触发条件满足
  │
  ▼
┌─────────────────────┐
│ Step 1: Judge       │──── LLM 判断这段对话值不值得记

└────────┬────────────┘
         │ 不值得记 → 更新状态 → 跳过
         ▼ 值得记
┌─────────────────────┐
│ Step 2: 写日记       │──── LLM 以第一人称写叙事
│ _write_diary()      │     输出：日记正文（不含 frontmatter）
└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│ Step 3: 组装         │──── 用 build_diary_content() 包装 frontmatter
│ build_diary_content()│     {date, mood, importance, topics}
└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│ Step 4: 存日记       │──── diary_store.append() → 返回 diary_id
│ diary_store.append()│
└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│ Step 5: 提原子       │──── LLM 从日记正文提取关键事实

         ▼

│   ensure_fact(content)              │

│   link_fact(diary_id, fact_id)      │

└────────┬────────────────────────────┘
         │
         ▼
┌─────────────────────┐
│ Step 7: 更新图谱     │──── 解析 content 中的 [[实体]]

└────────┬────────────┘
         │
         ▼
┌─────────────────────┐

         ▼
┌─────────────────────┐
```


### 4.5 重要度衰减（后台任务）

```
启动时注册后台循环：
  while True:
    await sleep(24h)

      atomic_facts.importance *= 0.99

  长期不提 → 0.95 → 0.94 → 0.93 → ...
```

## 五、存储设计详解

### 5.1 连接池

所有 Store 共享同一个 SQLite 连接池：

```
BaseDbStore._pool
  └── {db_path: {loop_id: {conn, lock, last_used}}}

- 按 (db_path + event_loop) 缓存
- 最多 24 个连接，LRU 淘汰
- 每个连接配一个 asyncio.Lock 防并发冲突
```

### 5.2 数据库文件

```
plugin_data/Memory/memory.db
  ├── 日记 + 原子 + 图谱 全部在一个文件
  ├── WAL 模式（PRAGMA journal_mode=WAL）
  └── FTS5 全文索引
```

### 5.3 版本迁移

```
db_version 表记录当前 schema 版本
v1: 初始列补齐
v2: 会话表
v3: schema 整合 + 索引兜底
v4（当前）: atomic_facts + diary_fact_links
```

迁移前自动备份到 `migration_backups/` 目录。

## 九、已知限制

1. **旧数据 user_id 不统一**：livingmemory 导入的数据分布在 `Hana`、`default`、`2398604399` 等多个 ID 下
2. **旧原子无 diary_id**：`/记忆重构` 已处理大部分，但旧数据的 `diary_id` 匹配可能不精确
3. **LLM 重要度偏高**：提取原子时 LLM 倾向于给 0.85~0.95，衰减需要一段时间才能平衡
4. **群聊多用户区分**：已实现 sender_id/sender_name 存储，但同名检测和合并尚未做
